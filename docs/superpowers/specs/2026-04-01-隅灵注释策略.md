# 隅灵桌宠注释策略（2026-04-01）

## 目标
- 让第一次接触项目的开发者无需阅读上下文即可理解关键逻辑。
- 确保跨端（Rust + Vue3 + Pinia）注释风格统一，后续开发遵循同一约定。
- 在保持代码可维护性的前提下满足“尽可能详细”的注释需求。

## 总体原则
1. **模块说明优先**：每个文件顶部必须写出模块职责、对外能力以及与其他模块的依赖关系。
2. **函数块注释**：凡是导出函数、Pinia action、生命周期钩子或 Tauri 命令，都需写多行注释描述参数、返回值与副作用。
3. **重点行内注释**：复杂算法 / 状态机 / 操作系统交互需要逐行或逐语句附注，包括拖拽坐标、动画循环、Rust 轮询等。
4. **语义而非语法**：注释描述“为什么这样做”或“输入输出是什么”，不赘述赋值语句本身。
5. **同步更新**：修改逻辑时若导致注释失效，必须同步调整；评审时如果发现注释滞后，视为阻塞问题。

## 实施范围
| 范围 | 举例 | 注释密度 |
| --- | --- | --- |
| 基础壳层 | `src/main.js`, `src/platformBridge.js` | 模块 + 函数说明，必要时段落注释 |
| 交互核心 | `src/App.vue`, `src/motion/useMotionController.js`, `src/stores/petStore.js` | 模块 + 函数 + 关键语句逐行 |
| 后端桥接 | `src-tauri/src/lib.rs` | 模块 doc comment + struct/命令 doc comment + 关键流程行注 |
| 资源文件 | `manifest.json` 等无法嵌入注释的文件，需要以文档说明字段含义 | 通过 README/设计文档补充 |

## 提交流程
1. 新功能开发前，先检查对应模块是否已有足够注释，必要时补充或重写。
2. 代码提交前自检：
   - `git diff` 中是否包含新增/更新的注释？
   - 核心逻辑（状态切换、系统调用）是否具备逐行注释？
   - 注释使用中文，描述准确。
3. Code Review 时把“注释缺失或过期”作为默认检查项，必要时打回。

## 后续工作
- 阶段三开始前，补充提醒引擎 API 文档，保持与本策略一致。
- 若后续引入 TypeScript 或多语言版本，需要更新本文件，说明不同语言的注释风格细节。
